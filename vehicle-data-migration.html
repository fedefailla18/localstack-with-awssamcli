🚗 Vehicle Data Migration Strategy

A safe & transparent approach to syncing with updated VCDB reference data

🔹 Project Context

  • Migrating from vcdb_vehicle_data ➜ vcdb_vehicle_data_new
  • consumer_car_details stores car info as values (not FKs)
  • Reference data evolved (makes, models, engines)

⚠️ The Challenge

  • No 1:1 mappings
  • Changes like "EQB 250" ➜ "EQB 250+"
  • Impacts pricing, API logic, rules

🧠 Our Process

  1. Detect outdated values (via EXISTS/NOT EXISTS)
  2. Suggest replacements (with similarity score)
  3. Manual confirmation
  4. Store in temp_car_mappings
  5. Apply safe updates to consumer data

✅ Benefits

  • Accurate: validated replacements only
  • Safe: controlled updates
  • Transparent: all mappings logged
  • Repeatable: supports future updates

📈 Next Steps

  • Finalize mappings (make, model, trim, engine_id, engine_desc)
  • Run update scripts
  • Validate downstream systems
  • Prep automated pipelines for future refreshes
